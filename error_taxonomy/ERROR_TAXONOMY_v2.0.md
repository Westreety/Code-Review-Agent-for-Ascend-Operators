## 一类：参数校验类错误

> 注入层：op_api。注入目标：`CheckMulParams()` / `CheckMulsParams()` / `CheckInplaceMulParams()` / `CheckInplaceMulsParams()`
> 严重度说明：★★★ 高危（静默放过非法输入，后续可能导致崩溃或错误结果）

### 1.1 — 空指针校验缺失

**注入位置**：`CheckMulNotNull()`（`aclnn_mul.cpp` 第140-145行）

**注入方式**：删除 `out` 的空指针检查。

**后果**：传入 `out=nullptr` 时，函数返回 `true`，后续对 `out` 解引用致段错误（SEGFAULT）。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🟢 易

---

### 1.2 — dtype 白名单遗漏（合法 dtype 误杀）

**注入位置**：`ASCEND910B_DTYPE_DTYPE_SUPPORT_LIST`

**注入方式**：删除 `DT_DOUBLE`，导致 DOUBLE 类型合法输入被 API 层误杀。

**严重度**：★★☆ | **注入难度**：🟢 易 | **可检测性**：🟢 易

---

### 1.3 — dtype 白名单过宽（非法 dtype 放行）

**注入位置**：`ASCEND910B_DTYPE_DTYPE_SUPPORT_LIST`

**注入方式**：添加 `DT_UINT32`，非法输入通过校验后崩溃或未定义行为。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🟡 中

---

### 1.4 — 错误码伪装

**注入位置**：`CheckMulParams()`

**注入方式**：将错误码 `ACLNN_ERR_PARAM_NULLPTR` 改为 `ACLNN_SUCCESS`。校验失败时返回成功，后续崩溃。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🟢 易

---

### 1.5 — API 变体间校验逻辑不一致

**注入方式**：对某一个 API 变体遗漏某项校验，但对另一个变体保留了。

**后果**：同样的非法输入，不同 API 变体行为不一致。

**严重度**：★★☆ | **注入难度**：🟡 中 | **可检测性**：🟡 中

### 1.6 — 格式检查缺失/错误

**注入方式**：删除格式检查逻辑，非 ND 格式 tensor 静默通过。

## 六类：计算执行类错误

> 注入层：op_kernel。注入目标：`mul_apt.cpp`（内核派发）+ `mul_dag.h`（8条计算流水线）+ `mul_struct.h`（tiling key模板）
>
> **8 条流水线**（mul_dag.h）：
> - `MulInt8Op`：INT8→INT32→Mul→AndFF(255)→uint8→CopyOut(int8)
> - `MulUint8Op`：UINT8→UINT32→Mul→AndFF→uint8
> - `MulBoolOp`：bool(int8)→half→Mul→ROUND→int8
> - `MulXfp16Op`：half/bf16→float→Mul→RINT→原类型
> - `MulComplex32Op`：complex32→complex64→Mul→RINT→complex32
> - `MulOp`：float/int32/int64/int16/complex64 → 原生 Vec::Mul
> - `MulDoubleOp`：手动软件实现（120行位操作，处理Inf/NaN/Zero/Subnormal）
> - `MulMixFpOp`：异构精度→promote→Mul→CopyOut

### 6.1 — 整数溢出未处理

**注入方式**：删除 INT8 饱和截断（AndFF），127×2=-2 而非 127。

**严重度**：★★★ | **注入难度**：🔴 难 | **可检测性**：🟡 中

---

### 6.2 — 特殊浮点值传播错误

**注入方式**：删除 NaN 检查或 Inf×0=NaN 规则，违反 IEEE 754。

**严重度**：★★★ | **注入难度**：🔴 难 | **可检测性**：🔴 难

---

### 6.3 — Double 精度实现错误

**注入方式**：手写 IEEE 754 乘法中指数偏移 ±1，边界值错误。

**严重度**：★★☆ | **注入难度**：🔴 难 | **可检测性**：🔴 难

---

### 6.4 — `if constexpr` 分支顺序错误（编译期类型分发错误）

**注入位置**：`mul_apt.cpp` 的 `if constexpr` 链

**注入方式**：调换 `MulOp` 和 `MulXfp16Op` 两个分支的顺序。half 匹配到 `MulOp`（直接硬件乘法）而非 `MulXfp16Op`（float32 提升）。

**严重度**：★★☆ | **注入难度**：🟢 易 | **可检测性**：🔴 难

---

### 6.5 — 跨层 dtype 声明不一致

**注入位置**：`mul_proto.h` 声明的 dtype 与 `mul_def.cpp` 注册的组合不一致（源码既有问题）

**严重度**：★★☆ | **注入难度**：—（源码本身存在） | **可检测性**：🟡 中

---

### 6.6 — Cube/Vector 中间精度截断

**注入位置**：`mul_dag.h` 中各计算流水线的类型转换路径

**描述**：AscendC 的 Cube 单元（矩阵乘）输出通常为 FP32 累加，但写回 UB 时可能被截断为 FP16/BF16。Vector 单元的中间结果在寄存器中可能保持更高精度，最终写回时截断。如果截断时机不对（如应在累加完成后再截断，却提前截断了中间结果），会导致精度损失。

**注入方式**：在 `MulXfp16Op` 或 `MulMixFpOp` 的流水线中，将 `RINT`（舍入）操作提前到中间结果而非最终输出，导致累积误差。

**严重度**：★★☆ | **注入难度**：🔴 难 | **可检测性**：🔴 难

**严重度**：★☆☆ | **注入难度**：🟢 易 | **可检测性**：🟢 易

### 7.4 — L1/UB 分配超限与碎片化

**注入位置**：`mul_dag.h` 或自定义 kernel 的 `AllocLocalTensor` 调用

**描述**：UB（Unified Buffer）和 L1 是 AI Core 的片上存储，大小固定（910B 的 UB=192KB）。手写流水线时如果 `AllocLocalTensor` 超出 UB 容量，或者双缓冲 ping-pong 分配导致碎片化，会触发硬件异常。现有 5.3 只覆盖了 tiling 阶段对 UB 的静态估算，不覆盖 kernel 内部的动态分配。

**注入方式**：在双缓冲 ping-pong 流水线中，多分配一个不必要的 buffer，使总分配量刚好超出 UB 容量。

**严重度**：★★★ | **注入难度**：🟡 中 | **可检测性**：🔴 难

## 八类：非确定性计算错误（Heisenbug）

> ⚠️ 本类是针对真实 NPU 异构计算环境中**最难捕获的缺陷**新增的分类。前 7 类均为确定性错误（相同输入→相同输出），本类的核心特征是 **100 次运行中仅 1~5 次触发**。

### 8.1 — 同步屏障缺失（AI Core 内流水线并发竞争与异步 DMA 数据竞争）

**对应场景**（微观层级）：单个 AI Core 内部，DMA 搬运单元（MTE1/MTE2/MTE3）和 Vector 计算单元是独立且异步工作的硬件模块。若 Kernel 代码漏写 `SyncAll` 或 `EnQue` 同步屏障，DataCopy 的异步 DMA 操作尚未完成时 Vector 单元就开始读取数据，导致数据竞争。

> ⚠️ **实测结论（2026-06-11）**：在 Ascend C 框架中，`DataCopy(Global→Local)` 和 `VecMul` 之间缺失 `SyncAll` 不是偶发性 bug，而是确定性崩溃。该错误类型在 Ascend C 框架中不存在"偶发性"的可能。但在更底层的裸指令编程或非 Ascend C 框架的场景中，同步屏障缺失仍可能表现为偶发性竞争。本条目保留作为分类参考。

**严重度**：★★★ | **注入难度**：🟡 中（代理） | **可检测性**：🔴 难 | **偶发性**：❌ 已验证非偶发

### 8.2 — 脏内存/未初始化内存

**对应场景**：NPU 的 `LocalTensor` 分配后不自动清零，上一任算子遗留的垃圾数据影响当前算子。

**严重度**：★★★ | **注入难度**：🟡 中（代理） | **可检测性**：🔴 难

### 8.3 — Tiling 分块重叠与跨核边界写污染（中观层级：多核内存踩踏）

**对应场景**（中观层级）：一张 NPU 芯片包含 30+ 个 AI Core。算子启动后输入被 tiling 切分，分发到多 core 并行处理。若分块边界计算错误，两个 core 同时写 Global Memory 的同一区域，或相邻 core 的输出因边界计算偏移而互相越界写入。

```
正确分块：
  Core 0 写 GM[0..511]    Core 1 写 GM[512..1023]    互不干扰 ✓

Buggy分块（offset 错位）：
  Core 0 写 GM[0..511]    Core 1 写 GM[508..1019]    ← 重叠 [508..511]！
  最后写的那一个覆盖前一个 → 重叠区域结果非确定性

跨核边界写污染（子场景）：
  Core 0 写 GM[0..515]    Core 1 写 GM[508..1023]    ← Core 0 越界写了 Core 1 的区域
  且 Core 1 的部分结果被 Core 0 的脏数据覆盖
```

**注入方式**：`extraSize = -1` 或 `extraSize = 0`（实际需要 DCACHE_SIZE=32KB），或 offset 计算中使用错误的 blockLen。

**测试攻击点**：大 tensor `[1024, 1024]` Double → 至少 2 个 core 参与 → 跑 100 次，比对重叠边界元素的数值 → 期望精确一致，buggy 版在 ±1 ULP 到完全错误之间抖动。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🔴 难

### 8.4 — 原子操作缺失

**对应场景**：多 core 对同一内存地址做读-改-写操作时，未使用原子指令。

**在 Mul 算子中的体现**：当前 Mul 的实现每个 core 独自分块、独立写回 GM，天然避免了共享写入。

**严重度**：★★★ | **注入难度**：—（当前代码无此模式） | **可检测性**：—（需交叉验证）

### 8.5 — 内存序问题（CPU-NPU 一致性）

**对应场景**：CPU 侧调用 `aclrtSynchronizeStream` 之后，NPU 写回的数据在 CPU cache 中未刷新。

**严重度**：★★★ | **注入难度**：🟡 中（代理） | **可检测性**：🔴 难

---

### 8.6 — 流水线同步死锁

**对应场景**：双缓冲 ping-pong 流水线中，`SetQueue`/`EnQue`/`DeQue` 的读写顺序错误，或者 buffer 切换逻辑有误，导致两个 pipe stage 互相等待。例如 stage 0 等待 stage 1 释放 buffer，stage 1 等待 stage 0 写入数据，形成循环依赖。

**与 8.1 的区别**：8.1 是"缺同步"（race），8.6 是"同步错误导致死锁"——sync 指令存在但使用方式导致互相等待。

**注入方式**：在 ping-pong 流水线中，交换 EnQue 和 DeQue 的顺序，或使用错误的 buffer index。

**严重度**：★★★ | **注入难度**：🔴 难 | **可检测性**：🔴 难

---

### 8.7 — 内存对齐与 Padding 残留污染

**对应场景**：NPU 的 DMA 搬运要求对齐（如 32B、512B 等），不对齐时会有 padding 字节被填充到对齐边界。如果 padding 区域是脏数据（未被显式写入），被误写入输出 tensor 的尾部，就会产生非确定性的脏值。

**与 8.2 的区别**：8.2 是 UB 分配后未清零导致读入脏数据；8.7 是 DMA 搬运时 padding 字节的脏数据被写出到 GM，污染输出。

**注入方式**：构造非对齐的 tensor shape（如 13 个 float），使得输出 tensor 的尾部有 padding 字节。

**严重度**：★★☆ | **注入难度**：🔴 难 | **可检测性**：🔴 难

## 九类：分布式与集群通信错误

> 注入层：跨算子/跨节点。Mul 算子通常嵌入在更大的计算图中，前后可能有 AllReduce 等集合通信操作。

### 9.1 — 算子流与通信流同步缺失

**对应场景**：计算流（Compute Stream）和通信流（Communication Stream）是独立的。如果算子执行后未在通信流前插入同步事件，通信操作可能读到未完成的中间结果。

**注入方式**：在分布式测试中，移除 `aclrtStreamWaitEvent` 或 `aclrtSynchronizeStream` 调用。

**严重度**：★★★ | **注入难度**：🔴 难 | **可检测性**：🔴 难

---

### 9.2 — 跨节点 Shape/Type 不一致死锁

**对应场景**：不同节点的算子输入 shape 或 dtype 不一致时，AllReduce 等集合通信操作可能因不匹配而阻塞，导致分布式训练 hang。

**注入方式**：在分布式测试中，构造不同 rank 的输入 shape 差一维。

**严重度**：★★★ | **注入难度**：🔴 难 | **可检测性**：🔴 难

---

### 9.3 — 通信域（Communicator）资源竞争

**对应场景**：多个算子同时使用同一通信域时，资源竞争可能导致部分通信操作被跳过或超时。

**注入方式**：在分布式测试中，同时发起多个通信操作而不进行流管理。

**严重度**：★★☆ | **注入难度**：🔴 难 | **可检测性**：🔴 难

---

### 9.4 — 集合通信精度截断错误

**对应场景**：集合通信操作（如 AllReduce）在跨节点传输时，可能将数据截断为低精度格式（如 FP32→FP16）以节省带宽，导致精度损失。

**注入方式**：在分布式测试中，使用需要高精度的输入（如 FP32 的大数值），观察跨节点后精度是否下降。

**严重度**：★★☆ | **注入难度**：🔴 难 | **可检测性**：🔴 难
