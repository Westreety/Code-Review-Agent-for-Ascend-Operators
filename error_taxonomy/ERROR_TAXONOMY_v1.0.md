## 一类：参数校验类错误

> 注入层：op_api。注入目标：`CheckMulParams()` / `CheckMulsParams()` / `CheckInplaceMulParams()` / `CheckInplaceMulsParams()`
> 严重度说明：★★★ 高危（静默放过非法输入，后续可能导致崩溃或错误结果）

### 1.1 — 空指针校验缺失

**注入位置**：`CheckMulNotNull()`（`aclnn_mul.cpp` 第140-145行）

**注入方式**：删除 `out` 的空指针检查。

**后果**：传入 `out=nullptr` 时，函数返回 `true`，后续对 `out` 解引用致段错误（SEGFAULT）。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🟢 易

---

### 1.2 — dtype 白名单遗漏（合法 dtype 误杀）

**注入位置**：`ASCEND910B_DTYPE_DTYPE_SUPPORT_LIST`

**注入方式**：删除 `DT_DOUBLE`，导致 DOUBLE 类型合法输入被 API 层误杀。

**严重度**：★★☆ | **注入难度**：🟢 易 | **可检测性**：🟢 易

---

### 1.3 — dtype 白名单过宽（非法 dtype 放行）

**注入位置**：`ASCEND910B_DTYPE_DTYPE_SUPPORT_LIST`

**注入方式**：添加 `DT_UINT32`，非法输入通过校验后崩溃或未定义行为。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🟡 中

---

### 1.4 — 错误码伪装

**注入位置**：`CheckMulParams()`

**注入方式**：将错误码 `ACLNN_ERR_PARAM_NULLPTR` 改为 `ACLNN_SUCCESS`。校验失败时返回成功，后续崩溃。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🟢 易

---

### 1.5 — API 变体间校验逻辑不一致

**注入方式**：对某一个 API 变体遗漏某项校验，但对另一个变体保留了。

**后果**：同样的非法输入，不同 API 变体行为不一致。

**严重度**：★★☆ | **注入难度**：🟡 中 | **可检测性**：🟡 中

---

### 1.6 — 格式检查缺失/错误

**注入位置**：`MulsCheckFormat()` / `MulCheckFormat()`

**注入方式**：删除格式检查逻辑，非 ND 格式 tensor 静默通过。

**严重度**：★★☆ | **注入难度**：🟢 易 | **可检测性**：🟡 中

## 二类：类型推导/提升类错误

> 注入层：op_api。注入目标：`InferTensorScalarDtype()` / `IsMulMixDtypeSupport()` / `CombineCategoriesWithComplex()` / `CheckMulPromoteType()`

### 2.1 — 混合精度组合被误删

**注入位置**：`IsMulMixDtypeSupport()`

**注入方式**：删除 BF16 相关的两条，BF16-FLOAT 组合不走优化路径。

**严重度**：★★☆ | **注入难度**：🟡 中 | **可检测性**：🟡 中

---

### 2.2 — Complex 类型推导回退错误

**注入位置**：`InnerTypeToComplexType()`

**注入方式**：BF16→COMPLEX64 映射改为返回 `DT_UNDEFINED`，类型推导链断裂。

**严重度**：★★☆ | **注入难度**：🟡 中 | **可检测性**：🟢 易

---

### 2.3 — Tensor×Scalar 精度保持判断丢失

**注入位置**：`InferTensorScalarDtype()`

**注入方式**：删除 `IsFloatEqual` 精度保持判断，无条件提升到 FLOAT，引入精度误差。

**严重度**：★★☆ | **注入难度**：🟡 中 | **可检测性**：🔴 难

---

### 2.4 — 类型转换可逆性检查缺失

**注入位置**：`CheckMulsPromoteDtype()`

**注入方式**：删除 `inferDtype→out` 的类型转换检查，非法类型转换放行导致数据损坏。

**严重度**：★★★ | **注入难度**：🟡 中 | **可检测性**：🟡 中

**注入方式**：删除 `IsDoubleSupport` 条件或 AiCore 支持列表加不支持 dtype，导致算子路由到错误设备。

**严重度**：★★☆ | **注入难度**：🟢 易 | **可检测性**：🟢 易

### 3.3 — 非连续内存优化路径错误

**注入方式**：强制走 Contiguous 路径（性能退化）或 `IsMulSupportNonContiguous` 对不支持情况返回 true（功能 bug）。

**严重度**：★★☆ | **注入难度**：🔴 难 | **可检测性**：🔴 难（性能）/ 🟡 中（结果错误）

## 八类：非确定性计算错误（Heisenbug）

> ⚠️ 本类是针对真实 NPU 异构计算环境中**最难捕获的缺陷**新增的分类。前 7 类均为确定性错误（相同输入→相同输出），本类的核心特征是 **100 次运行中仅 1~5 次触发**。

### 8.1 — 同步屏障缺失（AI Core 内流水线并发竞争）

**对应场景**（微观层级）：单个 AI Core 内部，DMA 搬运单元和 Vector 计算单元异步工作。若 Kernel 代码漏写 `SyncAll` 同步屏障。

> ⚠️ **实测结论（2026-06-11）**：在 Ascend C 框架中，`DataCopy(Global→Local)` 和 `VecMul` 之间缺失 `SyncAll` 不是偶发性 bug，而是确定性崩溃。该错误类型在 Ascend C 框架中不存在"偶发性"的可能。本条目保留作为分类参考。

**严重度**：★★★ | **注入难度**：🟡 中（代理） | **可检测性**：🔴 难 | **偶发性**：❌ 已验证非偶发

### 8.2 — 脏内存/未初始化内存

**对应场景**：NPU 的 `LocalTensor` 分配后不自动清零，上一任算子遗留的垃圾数据影响当前算子。

**严重度**：★★★ | **注入难度**：🟡 中（代理） | **可检测性**：🔴 难

### 8.3 — Tiling 分块重叠（中观层级：多核内存踩踏）

**对应场景**（中观层级）：一张 NPU 芯片包含 30+ 个 AI Core。算子启动后输入被 tiling 切分，分发到多 core 并行处理。若分块边界计算错误，两个 core 同时写 Global Memory 的同一区域。

```
正确分块：
  Core 0 写 GM[0..511]    Core 1 写 GM[512..1023]    互不干扰 ✓

Buggy分块（offset 错位）：
  Core 0 写 GM[0..511]    Core 1 写 GM[508..1019]    ← 重叠 [508..511]！
  最后写的那一个覆盖前一个 → 重叠区域结果非确定性
```

**注入方式**：`extraSize = -1` 或 `extraSize = 0`（实际需要 DCACHE_SIZE=32KB）。

**测试攻击点**：大 tensor `[1024, 1024]` Double → 至少 2 个 core 参与 → 跑 100 次，比对重叠边界元素的数值 → 期望精确一致，buggy 版在 ±1 ULP 到完全错误之间抖动。

**严重度**：★★★ | **注入难度**：🟢 易 | **可检测性**：🔴 难

### 8.4 — 原子操作缺失

**对应场景**：多 core 对同一内存地址做读-改-写操作时，未使用原子指令。

**在 Mul 算子中的体现**：当前 Mul 的实现每个 core 独自分块、独立写回 GM，天然避免了共享写入。

**严重度**：★★★ | **注入难度**：—（当前代码无此模式） | **可检测性**：—（需交叉验证）

### 8.5 — 内存序问题（CPU-NPU 一致性）

**对应场景**：CPU 侧调用 `aclrtSynchronizeStream` 之后，NPU 写回的数据在 CPU cache 中未刷新。

**严重度**：★★★ | **注入难度**：🟡 中（代理） | **可检测性**：🔴 难
